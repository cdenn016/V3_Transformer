## Finding 1: Validation and test loaders inherit training shuffle and drop the tail

Severity: high for metrics and model selection.

`vfe3/data/datasets.py:126-148` defines `make_dataloader(..., shuffle=True)` and always returns `DataLoader(..., shuffle=shuffle, drop_last=True)`. `train_vfe3.py:267-268` builds both train and validation loaders through `_select_loader(...)` with no split-specific override, and `train_vfe3.py:305-306` builds the held-out test loader the same way. This is not a concern about a chosen batch size. It is split semantics: validation and test evaluation should be stable corpus measurements, but the current loader shuffles them and discards any partial final batch.

The failure mode is sharper because `vfe3/train.py:222-264` computes `ce = total_nats / max(total_tok, 1)` and then `ppl = exp(min(ce, 20))`. If a validation or test split produces fewer windows than `batch_size`, `drop_last=True` yields no batches, `total_tok` remains zero, and evaluation reports `ce = 0` and `ppl = 1`. Even when the split is larger, a non-divisible split silently omits a random tail because shuffling is still enabled. The evaluation docstring at `vfe3/train.py:237-240` says the aggregation includes a partial last batch, so the implementation and its own stated contract disagree.

The fix should be split-aware at loader construction. Training can keep `shuffle=True, drop_last=True`; validation and test should use `shuffle=False, drop_last=False`. Add a regression test that constructs a tiny cached split with fewer windows than `batch_size` and asserts evaluation consumes tokens rather than returning the artificial perfect metric.

## Finding 2: The free-energy descent artifact mixes incompatible domains and reductions

Severity: medium for theory-facing artifacts.

`vfe3/train.py:396-419` logs validation CE from `evaluate(model, val_loader, ...)`, but it obtains `self_coupling`, `belief_coupling`, `attention_entropy`, and `free_energy_total` from `model.diagnostics(tokens)` on the live training batch. `vfe3/model/model.py:676-680` delegates those terms to `metrics.free_energy_terms(...)`, and `vfe3/metrics.py:124-132` returns sums over the selected diagnostic sequence. The artifact path then feeds those training-batch sums plus validation mean CE into `figs.plot_free_energy_descent` at `vfe3/run_artifacts.py:270-280`.

The result is not a closed free-energy decomposition over one empirical object. The F terms are summed diagnostics from one train batch, while `val_ce` is a token-weighted mean over validation data. There is a second mismatch in `vfe3/viz/figures.py:404-416`: panel A stacks `self_coupling`, scaled belief coupling, scaled attention entropy, and `val_ce`, but panel B uses `history["free_energy_total"]` when present, and that value came from `metrics.free_energy_terms` without the CE data term. So one panel includes the CE term and the other can plot a total that excludes it.

The clean repair is to compute all plotted components on the same split and with the same reduction. If the artifact is meant to be descriptive rather than a literal decomposition, it should be relabeled and separated from the "closes to runtime F" claim. If it is meant to close the canonical stack, the data term should be accumulated on the same validation stream and the F terms should be accumulated with matching token normalization or matching sums.

## Finding 3: Learned `lambda_beta` is logged but ignored by the free-energy descent stack

Severity: medium for learned-coupling runs.

When `learnable_lambda_beta=True`, `vfe3/train.py:416-418` writes the current `exp(log_lambda_beta)` into each history row. `vfe3/run_artifacts.py:257-263` already plots that trajectory. But `vfe3/run_artifacts.py:278-280` calls `plot_free_energy_descent(hist, lambda_beta=getattr(cfg, "lambda_beta", 1.0), ...)`, and `vfe3/viz/figures.py:407-408` scales belief coupling and attention entropy by the supplied `lambda_beta`. In a learned run, the artifact therefore scales the stack by the static config scalar rather than the recorded learned trajectory.

The local fix is small: pass the row-wise `hist["lambda_beta"]` vector when all free-energy rows contain it, falling back to the config scalar only on constant-coupling runs. This does not change training, but it does change what the artifact claims about the learned free-energy trajectory.

## Finding 4: Several registry-backed seams are still blocked by hardcoded config lists

Severity: medium for the project modularity contract.

`vfe3/config.py:12-24` defines static `_VALID_*` tuples. Some seams already validate against registries, such as divergence families at `vfe3/config.py:298-299`, covariance families at `vfe3/config.py:389-390`, transport/retraction/positional registries later in validation, and so on. Other seams do not. `vfe3/config.py:317` validates `gauge_group` against `_VALID_GAUGE_GROUPS` despite `vfe3/geometry/groups.py:57-71` exposing `register_group` and `get_group`. `vfe3/config.py:415` and `vfe3/config.py:447` validate attention priors against `_VALID_ATTENTION_PRIORS` despite `vfe3/attention_prior.py:20-32`. `vfe3/config.py:420` validates `alpha_mode` against `_VALID_ALPHA_MODES` despite `vfe3/alpha_i.py:19-46`. `vfe3/config.py:520` and `vfe3/config.py:535` validate decode and encode modes against static lists despite `vfe3/model/prior_bank.py:48-79`. `vfe3/config.py:566-567` does the same for norm selection despite `vfe3/geometry/norms.py:16-28`.

The decode seam is the most visible example. `vfe3/model/prior_bank.py:499-515` registers a `linear` decoder, and `PriorBank.decode` can route to it at `vfe3/model/prior_bank.py:238-239` when `use_prior_bank=False`, but `decode_mode="linear"` cannot pass config validation because `_VALID_DECODE_MODES` excludes it. That may be intentional as a second-gate design, but it still means the registry is not the authority for that seam.

This violates the project's stated extension rule: a variant should be added by writing and registering it, not by editing config call sites. The surgical fix is to add public registry-name functions, mirror `divergence_functionals()` and `divergence_families()`, and validate these fields against those registries. Keep intentional stubs, such as `encode_mode="gauge_fixed"`, as explicit post-registry guards.

## Finding 5: The optional model-channel coupling is predictively inert and tied to a detached flat transport

Severity: medium when `gamma_coupling > 0`; not a default-path bug.

The pure default path is preserved because `gamma_coupling` defaults to zero. Under `gamma_coupling > 0`, however, `vfe3/model/model.py:470-520` adds a model-channel reduced free-energy term only to the loss. It reads `s_mu, s_sigma = pb.encode_s(token_ids)` at `vfe3/model/model.py:500`, constructs transport with `compute_transport_operators(out.phi.detach(), self.group)["Omega"]` at `vfe3/model/model.py:502`, and adds `cfg.gamma_coupling * f_red_s.mean()` at `vfe3/model/model.py:520`. This route does not use the active `_transport` mode, does not consume `connection_W`, does not apply the active rope transport, and deliberately detaches `phi`. The logits and CE path remain byte-identical to `gamma_coupling=0` except for the regularization effect on the `s` tables.

That is acceptable as an opt-in regularizer if named and reported that way. It is not the full hierarchical `s -> q` model-coupling channel from the mathematical free-energy statement. If this path is used in theoretical artifacts, the code should either route through the active transport and make the phi-gradient choice explicit, or describe the term as a detached `s`-table regularizer with no direct predictive channel.

## Finding 6: The optional Regime-II learned connection is not constrained to be gauge-covariant

Severity: medium for non-flat, theory-facing runs; not a default-path bug.

The flat pure path exists and `connection_W` is not created unless `transport_mode="regime_ii"`. In that opt-in path, `vfe3/model/model.py:110-111` creates `connection_W` as an unconstrained `nn.Parameter` of shape `(n_gen, K, K)`. `vfe3/geometry/transport.py:177-181` then forms `delta_ij^a = mu_i^T W^a mu_j` by raw coordinate bilinear contraction and exponentiates the resulting generator coordinates into an edge factor.

The construction keeps the edge factor inside the group, but that is not the same as gauge covariance of the coefficient map. Under a change of belief coordinates, the source code does not transform `connection_W`, constrain it to an invariant tensor family, or enforce an intertwiner relation. This is a source-level inference from the implementation, checked against standard equivariance requirements in the cited group and gauge-equivariant literature. The code can still be useful as an opt-in non-flat relaxation, but claims of gauge-equivariance for trained nonzero `connection_W` need a proof or a property test that transforms both inputs and parameters consistently.

## Finding 7: Holonomy diagnostics use a deterministic low-index triangle prefix

Severity: low to medium for diagnostics quality and periodic overhead.

`vfe3/model/model.py:688` logs `metrics.holonomy_deviation(omega)`. That function, at `vfe3/metrics.py:50-91`, builds the first `max_triangles` distinct triples by nested loops in row-major order and therefore samples a fixed low-index prefix. A better estimator already exists in the same file: `vfe3/metrics.py:547-585` defines `holonomy_deviation_sampled`, with random distinct triples, bootstrap intervals, and a recorded span. The current default diagnostic is therefore both less representative and less informative than the available implementation.

The fix is to switch periodic diagnostics to the sampled estimator or log both the legacy scalar and the sampled mean/interval. This is not a hot training-path bug, but it matters because holonomy is one of the few observables used to interpret non-flat transport.

## Finding 8: `train_vfe3.py` reports a config-level tau that can be wrong for single-block groups

Severity: low, reporting only.

`train_vfe3.py:215` prints `tau={cfg.tau:.4f}`. `vfe3/config.py:602-614` explicitly says this property is a per-head convenience and that active attention temperature is computed by `free_energy.attention_tau(kappa, group.irrep_dims)`. The in-package `vfe3/train.py:446` uses the group-aware value. For single-block groups such as `glk`, `so_k`, or `sp`, the click-to-run banner understates the active temperature by `sqrt(n_heads)`. This is a reporting bug, and it matters because attention-temperature scaling is exactly the kind of quantity readers compare to the scaled-dot-product convention in Vaswani et al.

The `nn.Module` and `nn.Parameter` usage I saw is model plumbing or explicit opt-in parameterization rather than a hidden neural layer. The full-covariance decode and retraction paths remain computationally heavy, but they are opt-in pure paths rather than bugs.
