## Route B: gauge-COVARIANT non-flat Regime-II transport (new, default OFF)

New `transport_mode="regime_ii_covariant"`: a non-flat (curved, non-trivial-holonomy) connection
that, unlike `regime_ii` (`delta_ij = mu_i^T W mu_j`, gauge-invariant only at `W=0`), stays
gauge-COVARIANT (`Omega_ij -> g_i Omega_ij g_j^{-1}`) for ANY learned weight. The edge coefficients
are built from GAUGE-INVARIANT scalar features of the (query, flat-transported-key) belief pair:
`delta_ij^a = cocycle_relaxation * sum_f M^a_f I^f_ij` with `I^f = (Mahalanobis, trace, log-det)` of
`KL(q_i || Omega^0_ij q_j)`, `M` a learned `(n_gen, 3)` nn.Parameter (zero-init -> flat). Full GL(K)
(Route A — commutant-restricted W on a compact subgroup — left as a `TODO(Route A)` at the builder).

- `vfe3/geometry/transport.py`: `gauge_invariant_edge_features()` (the three KL-component invariants,
  Cholesky-based, exact GL(K)-invariant) and `@register_transport("regime_ii_covariant")`
  `_build_regime_ii_covariant()` (flat fast-path at M=None/0; soft-capped per-edge exp; self-edge
  excluded; same dict contract as flat).
- `connection_M` threaded model -> stack -> block -> e_step (`e_step_iteration`,
  `build_belief_transport`, `_transport`, `free_energy_value`, `phi_alignment_loss`); `sigma`/`sigma_key`
  read locally from the belief at the build sites (query live, key detached). `VFEModel.__init__`
  creates `connection_M` (zero-init) only under the covariant mode.
- `vfe3/gradients/kernels.py`: `uses_kernel_route` excludes `regime_ii_covariant` (the flat kernel
  drops dOmega/dmu) so it routes to the autograd oracle, like `regime_ii`.
- `vfe3/config.py`: registry-validated mode; the oracle-route / freeze / gaussian_full gates include
  the new mode. AUTO-ENABLES `oracle_unroll_grad=True` for BOTH non-flat regimes (the learned
  connection enters the loss only through the unrolled E-step) instead of only warning.
- `train_vfe3.py` / `ablation.py`: `transport_mode` toggle documents the new option (default `"flat"`
  = OFF); ablation `transport_mode` sweep gains a `regime_ii_covariant` arm.
- `tests/test_regime_ii_covariant.py` (new, 12 tests): feature GL(K)-invariance, builder flat-oracle
  (None/zero M) + non-flat holonomy + dict shape, model `connection_M` creation / init-flat==flat /
  nonzero-M-changes-forward / gradient-flows-to-M, and the auto-enable.

Causality note (decided): keep causal masking; "holonomy" lives on causal triangles k<j<i as
path-dependence (two-hop vs direct transport), so curvature is NOT vacuous under the causal mask
(only the naive directed-cycle reading is). In a single layer the curvature is latent (only direct
transports are applied) — its live effect is the per-edge delta reshaping every logit/value.

## Route B follow-up: OOM fix (query-chunking) + diagnostics threading of connection_M

Addressed the K=20 / 2-head `regime_ii_covariant` OOM and wired `connection_M` into the diagnostic
transports (the follow-up flagged above).

OOM root cause: at `n_e_steps=1` it is a SINGLE build's footprint, not an unroll multiplier. The
covariant builder held ~7 dense `(B,N,N,K,K)` tensors live at once (flat cocycle `omega0`, transported-
key covariance `cov_kt` + `cov_kt+eps`, its Cholesky `L_s`, `S^-1 Sigma_q`, `delta_mat`, `exp_delta`,
`omega`) -- ~1.68 GB EACH at B=64/N=128/K=20 -- and the phi-step build stacks a second on the oracle's
retained graph. Flat at K=80 never allocates a dense Omega (factored path), so the smaller-K covariant
run OOMs while the larger flat run fits.

- `vfe3/geometry/transport.py`: `_build_regime_ii_covariant` now loops over the QUERY index in chunks
  (`_regime_ii_query_chunk(B,N,K)`, capped at `_REGIME_II_CHUNK_ELEMS=64M` fp32 elems ~256 MB per
  `(B,chunk,N,K,K)` working tensor). The dense `(B,N,N,K,K)` flat cocycle / transported-key covariance
  / Cholesky factors / edge exp are never materialized whole. No cross-query reduction exists, so each
  `(i,j)` operator is identical to the one-chunk build (chunk>=N collapses to the original path bit-for-
  bit; B=1 diagnostic builds stay one chunk). Self-edge mask is rebuilt per chunk (diagonal column =
  `i0+local_row`). Pinned by `test_regime_ii_covariant_chunked_matches_unchunked` (forced size-1 chunk
  == one chunk for both Omega VALUES and the gradient to `connection_M`) + `test_regime_ii_query_chunk_*`.
- Diagnostics threading (the logged F-trajectory / holonomy_deviation / attention maps under
  `regime_ii_covariant` previously fell back to FLAT transport because only `connection_W` was passed):
  `connection_M` (+ `sigma` for the covariant features, + the extended `transport_mode in
  ("regime_ii","regime_ii_covariant")` mu-gate) now threaded into `VFEModel.diagnostics` / `attention_maps`
  and `viz/extract.py` (`_iter_kwargs`, `_fe_kwargs`, `belief_bank`, `e_step_belief_trace`,
  `numerical_health`, `converged_state`). New conditional diagnostic column `connection_m_norm` mirrors
  `connection_w_norm`. Pinned by `test_diagnostics_holonomy_reflects_connection_m` and
  `test_converged_state_omega_reflects_connection_m` (nonzero `connection_M` -> holonomy_deviation > 0,
  was ~7e-7 flat).

NOTE: `regime_ii` (bilinear W) was left unchunked (the user reported only the covariant case; W has no
covariance/Cholesky transients, so a lower peak). CUDA was unavailable on the dev box, so the peak-memory
reduction is verified by construction + numeric/grad-equivalence, not measured locally.
