## Head mixer auto-disables on a single head (config ergonomics)

`VFE3Config.__post_init__` (`vfe3/config.py`): `use_head_mixer=True` with a head-block group
(`block_glk`/`tied_block_glk`) and `n_heads < 2` no longer raises -- it now AUTO-DISABLES the mixer
(`self.use_head_mixer = False`) and emits a `UserWarning`. This is a single-HEAD scan artifact (one
gauge block, nothing to mix), so a head/K sweep can leave `use_head_mixer=True` set without a manual
toggle-off at `n_heads=1`. A genuinely single-BLOCK group (`glk`/`so_k`/`sp`) still RAISES -- choosing
a non-block group cannot mix regardless of head count, an independent static misconfiguration.

Tests (`tests/test_audit_fixes_2026_06_10.py`): `test_config_rejects_head_mixer_on_single_block_groups`
narrowed to the `glk` single-block raise; new `test_config_auto_disables_head_mixer_on_single_head`
pins the `block_glk`/`tied_block_glk` `n_heads=1` auto-disable + warning. Full suite: **1049 passed,
1 skipped, 1 xpassed, 0 failures/errors** (junit `tests=1051`).

## metrics.csv: validation columns are eval-cadence (blank between evals)

`metrics.csv` was carrying the most-recent validation FORWARD into every `log_interval` row, so
`val_ce/val_ppl/val_bpc`, `generalization_gap`, and the held-out probes repeated on every log line
between evals. Now they are written only on a step where the periodic eval actually ran; the
log-interval rows in between leave those cells BLANK -- matching VFE_2.0's `metrics.csv` (one
validation per `eval_interval`, empty otherwise). Train-side columns still log every `log_interval`.

`vfe3/train.py` (`train` do_csv block): `val_ce/ppl/bpc`, `generalization_gap`, and the
`_VAL_DIAG_KEYS` probes now write their value iff `do_eval`, else `float("nan")` (instead of
`last_val.get(...)` carry-forward). `vfe3/run_artifacts.py` (`log_metrics`): NaN cells render as an
empty string in the CSV file, while `self.history` keeps the raw NaN float so the figure pass
(`math.isfinite` filtering) is unchanged.

Test (`tests/test_report.py::test_metrics_csv_logs_at_log_cadence`): updated from the carry-forward
assertions to assert blank cells on non-eval rows and fresh values on eval rows. Affected suites
(report, run_artifacts, run_diagnostics, model_channel_diagnostics): **53 passed, 0 failures/errors**
(junit `tests=53`).

## New: `lambda_h_mode='state_dependent_per_coord'` (per-coordinate hyper-prior coupling)

Added the per-coordinate sibling of the hyper-prior coupling, mirroring `lambda_alpha_mode`'s
per-coord form. Each model-channel coordinate `k` gets its own envelope weight
`lambda_h^(k)* = c0_h^(k)/(b0_h^(k) + KL_k(s_i||r))` on the UNSUMMED per-coordinate divergence, with
`R_h^(k)` summed over `k`. TDD (`tests/test_lambda_h_per_coord.py`, 14 tests: registry, config guards,
model-integration envelope oracle, grad-flow on the scored and s_e_step routes).

- `vfe3/lambda_h_i.py`: `'state_dependent_per_coord'` added to `_LAMBDA_H_MODES`; new
  `lambda_h_is_per_coord(mode)` helper (validates membership, defers to `alpha_is_per_coord` â€” the
  per-coord flag has one source of truth since lambda_h delegates to the shared alpha registry).
  Docstring corrected (it had wrongly claimed "there is no per-coordinate form").
- `vfe3/config.py`: `b0_h`/`c0_h` typed `'float | List[float]'`; new validation mirroring alpha's
  per-coord guards keyed on `lambda_h_mode` â€” `(K,)` `b0_h`/`c0_h` list requires the per-coord mode +
  `len==embed_dim` + positivity; per-coord rejects full-covariance family and non-decomposable
  divergence (`squared_hellinger`). Field-doc inline keys updated.
- `vfe3/model/model.py`: `_hyper_prior_kl` gains `per_coord` (routes `self_divergence_per_coord` ->
  `(B,N,K)`); `_hyper_prior_weighted` applies the per-coord envelope, sums `R_h^(k)` over `K`, widens
  the R_h reg-gate to include the per-coord mode, and wraps `b0_h`/`c0_h` in `_as_coeff` for `(K,)`
  lists. The s-E-step site (`_refine_s`) also wrapped in `_as_coeff`; it otherwise inherits per-coord
  routing through the shared `lambda_alpha_mode` seam (zero `e_step` edits).

NOTE (verified, sympy + checkpoint measurement): this is NOT a width-robustness lever. Summing the
per-coordinate envelopes over `K` is linear in `K` (`~2.3x` K=50->120), unlike per-token
`state_dependent` (`log` in K, `1.15x`). It IS a genuine capability â€” measured per-coordinate
`KL(s||r)` is heterogeneous (within-token CV `~0.7` for frequent tokens; marginal `max/min ~23x`), so
the per-coordinate weight differentiates (near-prior coords pinned, far coords released) in a way the
single per-token weight cannot.

Full suite after the change: **1063 passed, 1 skipped, 1 xpassed, 0 failures** (165 warnings,
all pre-existing).

## New: two precision/Sigma diagnostic toggles (decode + attention, default OFF)

Motivation: the user finds the prior-bank KL decode underperforms the linear decode even at optimal
`decode_tau`. The likely reason is generative-vs-discriminative (Ng & Jordan 2002) -- the prior-bank
decode is a tied, capacity-starved generative/prototype classifier; the linear head is discriminative
and wins at this data scale -- which conflates "use Sigma" with "be generative". These two toggles
separate the questions: does Sigma carry predictive signal at the readout, and upstream in attention,
WITHOUT giving up the discriminative head. Both default OFF (pure path preserved) and TDD-pinned
(`tests/test_precision_toggles.py`, 12 tests).

- **`decode_precision_scaled`** (`vfe3/config.py`, `use_prior_bank=False` only): feed the
  precision-weighted mean -- the diagonal natural parameter `eta = Sigma^-1 mu = mu/(sigma+eps)` --
  to the linear head instead of the bare `mu`, so `Sigma_q` enters the DISCRIMINATIVE readout.
  `vfe3/model/prior_bank.py` `_decode_linear` applies it (and `PriorBank.__init__`/the `VFEModel`
  construction thread the flag); OFF -> `logits = mu_q @ W^T` byte-identical. Inert (warns) under
  `use_prior_bank=True` (KL decode already consumes `sigma_q`), mirroring `decode_bias`.

- **`precision_weighted_attention`** + **`precision_attention_b0`** (`vfe3/config.py`): fold a
  DETACHED per-key reliability bias `-log(b0 + tr Sigma_j)` into the attention `log_prior`, so
  attention down-weights high-variance (unreliable) keys before the softmax. New module-level
  `vfe3/model/model.py:_precision_key_bias`; `forward` folds it as a `(B,1,1,N)` term (broadcast over
  head + query) onto the position prior using the belief sigma entering the block (post s-refine).
  Detached -> the closed-form belief kernel treats it as a fixed prior (exact). A uniform-over-keys
  Sigma is softmax-absorbed (no effect); only key-to-key variance changes attention. OFF -> `log_prior`
  stays the sequence-only `(N,N)`/`(H,N,N)` bias, byte-identical. `__post_init__` requires `b0 > 0`.

Both toggles added to the `train_vfe3.py` config dict (default OFF). Full suite: **1075 passed,
1 skipped, 1 xpassed, 0 failures**.
