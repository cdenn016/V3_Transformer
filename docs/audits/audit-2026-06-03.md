### Root cause ‚Äî proven, framed neutrally

Nine of the ten failures share one cause: the config default `pos_phi` is **`"learned"`**
(`config.py:93`), but (a) its own comment one line above still says "BCH positional encoding
(**default-off**) ‚Ä¶ the pure path is `none`" (`config.py:89-92`), and (b) `test_pos_phi_defaults_off_and_validates`
asserts the default is `"none"` (`test_config.py:256`). `pos_phi="learned"` is default-ON, so a learned
positional-gauge parameter table is created for every default-config model, changing the forward and
the optimizer parameter groups ‚Äî which breaks the frozen-oracle pin, the byte-identity / copy-equivalence
tests, the linearity tests, the learnable-alpha test, and the optimizer-group test.

**Proven by experiment:** temporarily setting the default back to `pos_phi="none"` and re-running the
nine non-`test_viz` failures gives **9/9 passed** (the diagnostic edit was reverted; `config.py` is
back at `main`'s state). The flip rode in on commit `52162a5` (`chore(sync): local working tree ->
remote`).

The evidence (a `chore(sync)` commit, a comment that still says "default-off," and nine tests encoding
`"none"`) leans toward an **accidental** flip during the sync rather than a deliberate default change.
Either way, the default, the comment, and the suite disagree and `main` is red. This audit does **not**
pick the value (per the user's "don't worry about config values" directive). The two clean resolutions:

- **If accidental:** revert `config.py:93` to `pos_phi = "none"` ‚Äî restores green and matches the
  comment + tests.
- **If intended:** update the `config.py:89-92` comment (drop "default-off"), fix
  `test_pos_phi_defaults_off_and_validates`, and re-baseline the 8 forward-equivalence / byte-identity /
  optimizer-group tests against the `pos_phi="learned"` default.
