# GL(K) Attention Code Audit - 2026-05-30

## Scope

This audit covers the executable Python code under `vfe3/` and `tests/` against the theory claims in `Manuscripts-Theory/GL(K)_attention.tex`. I ignored config defaults as evidence of intended behavior. A finding is included only when a selected or advertised path is missing, unreachable, or mathematically different from the GL(K) attention path it purports to implement.

This was a local single-agent audit. The deep-audit skill normally dispatches subagents, but the available subagent tool may be used only when the user explicitly asks for subagents, so no subagent verifier was run.

## Theory Anchors

The manuscript defines gauge action on Gaussian beliefs as `(mu, Sigma) -> (Omega mu, Omega Sigma Omega.T)` and states KL invariance for invertible `Omega` in `Manuscripts-Theory/GL(K)_attention.tex:441` and `Manuscripts-Theory/GL(K)_attention.tex:522`. It derives `E_ij = KL(q_i || Omega_ij q_j)`, the entropy-regularized row objective, and the softmax stationary point in `Manuscripts-Theory/GL(K)_attention.tex:697`, `Manuscripts-Theory/GL(K)_attention.tex:721`, and `Manuscripts-Theory/GL(K)_attention.tex:752`. It then gives the temperature-scaled canonical alignment term and reduced free energy `-tau log Z_i` in `Manuscripts-Theory/GL(K)_attention.tex:766` and `Manuscripts-Theory/GL(K)_attention.tex:843`. The language-model algorithm presents per-head divergences and per-head attention weights in `Manuscripts-Theory/GL(K)_attention.tex:2060`.

## Confirmed Matches

The scalar free-energy assembly implements the canonical entropy-retained path and the entropy-suppressed surrogate as a toggle. `vfe3/free_energy.py:146` computes `beta`, `vfe3/free_energy.py:154` adds the belief-coupling term, and `vfe3/free_energy.py:157` gates the entropy term behind `include_attention_entropy`. The reduced free-energy helper normalizes raw log-prior biases through `torch.log_softmax` in `vfe3/free_energy.py:100`, so non-uniform priors do not leak an additive row-normalizer into `-tau log Z`.

The flat vertex transport path matches the Regime I cocycle in the manuscript. `vfe3/geometry/transport.py:84` embeds `phi` in the Lie algebra, `vfe3/geometry/transport.py:85` exponentiates `phi` and `-phi`, and `vfe3/geometry/transport.py:88` computes `Omega_ij = exp(phi_i) exp(-phi_j)`. The full-covariance transport helper also implements the sandwich action at `vfe3/geometry/transport.py:159`.

The full Gaussian KL kernel itself is GL(K)-invariant when used directly. Runtime probe:


This confirms the exact theorem for the full path and also shows why diagonal projection is not a pure GL(K) path under a non-orthogonal transform.

## Findings

### 1. Multi-head GL(K) attention collapses to a single shared beta

Severity: high.

The manuscript algorithm computes per-head divergences `D_ij^(h)` and per-head weights `beta_ij^(h)` in `Manuscripts-Theory/GL(K)_attention.tex:2062` through `Manuscripts-Theory/GL(K)_attention.tex:2064`. The current implementation can select `block_glk`, and `vfe3/geometry/groups.py:113` builds block generators with `irrep_dims = [d_head] * n_heads`, but the attention energy has no head axis. `vfe3/free_energy.py:25` through `vfe3/free_energy.py:36` returns one `(..., N, N)` energy tensor, and `vfe3/free_energy.py:70` through `vfe3/free_energy.py:81` returns one `(..., N, N)` beta tensor.

Runtime probe with `n_heads=3` and `irrep_dims=[2, 2, 2]`:

```text
n_heads 3
irrep_dims [2, 2, 2]
energy_shape (4, 4)
beta_shape (4, 4)
```

This means toggling `n_heads` changes the block structure of `Omega`, but it does not produce head-specialized attention distributions. The code is therefore a single attention distribution over a block-diagonal transport, not the per-head GL(K) attention algorithm described in the manuscript.

Fix: carry an explicit head axis through transport slicing, divergence, `attention_weights`, metrics, and visualization, or document the current implementation as single-beta block transport rather than multi-head GL(K) attention.

### 2. The full-covariance pure path is not runnable through the model or E-step

Severity: high.

The GL(K) invariance theorem requires full covariance push-forward `Omega Sigma Omega.T` in `Manuscripts-Theory/GL(K)_attention.tex:441` through `Manuscripts-Theory/GL(K)_attention.tex:457` and `Manuscripts-Theory/GL(K)_attention.tex:522` through `Manuscripts-Theory/GL(K)_attention.tex:564`. The language-model section states that the reported evolving covariance mode uses full covariance matrices in `Manuscripts-Theory/GL(K)_attention.tex:2030`.

The code has pieces of a full path, but the integrated model path remains diagonal. `vfe3/model/prior_bank.py:115` through `vfe3/model/prior_bank.py:117` create diagonal variance parameters, and `vfe3/model/prior_bank.py:183` through `vfe3/model/prior_bank.py:185` encode diagonal `sigma` tensors. `vfe3/inference/e_step.py:158` calls `retract_spd_diagonal` unconditionally after every covariance update. `vfe3/model/prior_bank.py:252` through `vfe3/model/prior_bank.py:266` registers `decode_mode="full"` but raises `NotImplementedError`.

Runtime probe with `family="gaussian_full"`:

```text
full_family_diagonal_decode RuntimeError The size of tensor a (3) must match the size of tensor b (4) at non-singleton dimension 0
full_family_full_decode RuntimeError The size of tensor a (3) must match the size of tensor b (4) at non-singleton dimension 0
```

Thus the exact GL(K) covariance path exists only as isolated kernels and tests, not as a selectable model/E-step path.

Fix: add full-covariance prior-bank parameters, encode full SPD beliefs, choose `retract_spd_full` when the selected family is full, implement full decode, and add an integration test that `VFEModel(VFE3Config(family="gaussian_full", decode_mode="full"))` runs.

### 3. `gauge_parameterization="omega_direct"` is a dead toggle in the model path

Severity: high.

`vfe3/config.py:53` exposes `gauge_parameterization`, and `vfe3/config.py:130` validates it. `vfe3/geometry/transport.py:92` through `vfe3/geometry/transport.py:132` implements a direct `Omega_i Omega_j^{-1}` transport capable of representing determinant-negative GL(K) elements. The model path does not use the toggle. `vfe3/model/prior_bank.py:117` always stores `phi_embed`, `vfe3/model/model.py:93` always builds beliefs from `phi`, and `vfe3/inference/e_step.py:27` through `vfe3/inference/e_step.py:32` always calls `compute_transport_operators`, not `compute_transport_operators_direct`.

The search result confirms that `gauge_parameterization` appears only in config validation under `vfe3/`, not in model, block, stack, or E-step dispatch.

Fix: either wire `cfg.gauge_parameterization` through the model and E-step with a real direct-Omega belief representation, or remove/reject the `omega_direct` option from the integrated model configuration until that path exists.

### 4. `divergence_family` is validated but unused by the executable model path

Severity: medium.

`vfe3/config.py:40` defines `divergence_family`, and `vfe3/config.py:111` validates it, but the only executable field passed into the E-step is `cfg.family` at `vfe3/model/block.py:37`. A repository search for `divergence_family` found only `vfe3/config.py` and `tests/test_config.py`.

For a user who toggles `divergence_family`, the model silently keeps using `cfg.family`. That is a dead seam, and it violates the project rule that config-selected registries sit behind each variant boundary.

Fix: collapse `family` and `divergence_family` into one field, or make `divergence_family` the field that is passed into `vfe_block`, `e_step`, `pairwise_energy`, and decode verification.

### 5. The phi mass term is an M-step loss regularizer, not part of the phi E-step

Severity: medium.

The manuscript algorithm includes the gauge-frame penalty inside the phi gradient objective in `Manuscripts-Theory/GL(K)_attention.tex:2077` through `Manuscripts-Theory/GL(K)_attention.tex:2079`. The code exposes `mass_phi` in `vfe3/config.py:63`, but `vfe3/model/block.py:30` through `vfe3/model/block.py:39` does not pass it into `e_step`, and `vfe3/inference/e_step.py:84` through `vfe3/inference/e_step.py:115` has no mass parameter in `phi_alignment_loss`. The only executable use is after the stack in `vfe3/model/model.py:117` through `vfe3/model/model.py:119`, where it is added to the training loss.

This is not wrong if `mass_phi` is intended solely as an outer training regularizer. It is wrong for the manuscript algorithm when `e_phi_lr > 0` and the user expects the phi E-step itself to descend the penalized objective.

Fix: either pass `mass_phi` into `phi_alignment_loss` and include `0.5 * mass_phi * ||phi||^2` during phi E-step updates, or rename/document the field as an M-step-only regularizer.

### 6. The filtered free-energy diagnostic freezes query frames when `keys` is supplied

Severity: low.

`vfe3/inference/e_step.py:70` selects `key_belief = belief if keys is None else keys`, and `vfe3/inference/e_step.py:71` builds the entire `Omega_ij` tensor from `key_belief.phi`. That freezes both source frames and query frames. A filtered objective with current query frames and frozen key beliefs cannot be represented by `free_energy_value(..., keys=...)`.

This is harmless for tests that keep `e_phi_lr=0` while checking filtered descent, but it becomes a diagnostic mismatch once phi is changing.

Fix: build transport from current query frames and frozen key frames when `keys` is supplied, or document that `keys` freezes the entire transport, not only the key-side beliefs.
