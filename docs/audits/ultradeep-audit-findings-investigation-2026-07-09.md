# Investigation of the Ultradeep Audit Findings — 2026-07-09

## Scope and Method

This document records the re-verification of all findings in `docs/audits/ultradeep-audit-most-recent-buildouts-2026-07-08.md` (findings 1–100, plus the seven unnumbered round-1 lows, tracked here as 101–107) against the current tree: branch `main` at commit `1b917d0` including the intentional uncommitted WIP in `ablation.py` and `train_vfe3.py`, which was treated as the live truth. Ninety-nine read-only subagents performed the verification: each of the eight audited-high findings received an adversarial skeptic, an independent defender, and a judge who spot-checked the decisive citations before ruling; each medium finding received a dedicated verifier; lows ran in batched verifiers. No source files were modified and no training, scaling, or full pytest runs were performed.

### Finding 2 — Scaling analysis can pool confounded routes and use inconsistent estimators

**Audited:** medium | **Verdict:** STILL_PRESENT | **Current severity:** medium | **Confidence:** high

Both halves of the finding are unchanged in the current tree. The pooled headline L(N) fit still excludes only the inference route and pools all other routes/arms (including the grow_K_mup fixed and mup arms and any token-budget differences), and the per-route table plus the routes-overlay figure still fit with the default unweighted no-offset log-log estimator while the headline pooled fit uses SEM-weighted Chinchilla offset form — so the alphas printed side by side in one report come from different estimators, and the figure's dashed "pooled" alpha can disagree with the console/summary pooled alpha of the same run. Partial mitigations the audit already implicitly credited do exist — the ANCOVA frontier-collapse F-test (scaling_analysis.py:250-285, 370-378) discloses route divergence, mixed data_sha256/git_sha warnings fire (318-323), and the bootstrap CI now reuses the headline weights/offset (204-239, fixing an earlier round's inconsistency) — but the pooled fit is reported unconditionally rather than gated on the F-test, and no per-route/overlay call reuses the headline estimator. This is analysis/reporting-layer only (no training impact) but is on the click-to-run scaling_analysis.py path under the current CONFIG (with_offset=True), so reported exponents feeding tables/figures are mutually inconsistent by construction.

**Evidence.** scaling_analysis.py:326 pools every non-inference route into one fit: `param_points = [p for p in points if p["route"] != INFERENCE_ROUTE]` (this includes both grow_K_mup arms and any token-budget-mismatched routes). The headline fit at scaling_analysis.py:345-348 uses WLS + offset: `w = np.where(sem > 0, (ys / ...) ** 2, 1.0); fit = _fit_power_law(xs, ys, weights=w, with_offset=CONFIG["with_offset"])` (CONFIG["with_offset"]=True at line 36). The per-route table at scaling_analysis.py:364-365 drops both: `fr = _fit_power_law(np.array([p["n_params"] for p in pr], ...), np.array([p["ce_mean"] for p in pr], ...))` — defaults weights=None, with_offset=False per vfe3/viz/figures.py:2344-2350. The overlay figure repeats this: vfe3/viz/figures.py:2503 `fit = _fit_power_law(st["x"], st["mean"])` and 2511 `g = _fit_power_law(stall["x"], stall["mean"])` (unweighted, no offset), while the headline figure at 2457 uses `weights=w, with_offset=with_offset`; plot_scaling_routes (2478-2486) has no with_offset parameter at all and _make_figures (scaling_analysis.py:477-479) passes none. Only post-audit change: commit 1b917d0 edited one line of scaling_analysis.py (input_dir "grow_K" -> "blocks_K48"); figures.py untouched.

**Recommended action.** Thread one estimator configuration through: compute per-point weights once, pass `weights` and `with_offset=CONFIG["with_offset"]` to the per-route fits at scaling_analysis.py:364-365; add a `with_offset` parameter (and optional SEM weighting) to plot_scaling_routes in vfe3/viz/figures.py and use it at lines 2503/2511, passing CONFIG["with_offset"] from _make_figures. Optionally annotate the pooled fit output with the F-test verdict (e.g. suppress or flag the pooled alpha when the ANCOVA says routes diverge) and warn when tokens_seen varies across pooled points.

### Finding 3 — Cost-model reporting misrepresents active linear-decode runs

**Audited:** medium | **Verdict:** STILL_PRESENT | **Current severity:** medium | **Confidence:** high

The finding is unchanged: `_cost_model_fields` still unconditionally counts the 2VK prior-bank decode readout in `active_params_per_token` and adds the V*K linear-projection weight on top under `use_prior_bank=False`, overstating the linear-decode working set by ~2VK, and `flops_per_token_estep` still omits the second s-channel E-step (`_refine_s`, a full `n_e_steps` e_step per forward) that runs under `s_e_step=True`. No commit since the audit touched run_artifacts.py. Under the CURRENT working-tree configs of both click-to-run entry points (`use_prior_bank=False`, `s_e_step=True`) every run's scaling_point is misstated, though this affects only reported cost/scaling bookkeeping, not training behavior. The `fpt_decode = 2VK` FLOP term happens to be roughly right for the linear matmul, so the defect is confined to `active_params_per_token` and the missing s-E-step FLOP term.

**Evidence.** vfe3/run_artifacts.py:621-625: `active = (2 * V * K) + (2 * K + n_gen)` then `if not cfg.use_prior_bank: active += V * K  # output_proj_weight (linear readout)` — the full-vocab prior-bank mu/sigma readout (2VK) is counted unconditionally and the linear-decode weight is ADDED on top, though the linear decode path (vfe3/model/prior_bank.py:863,872: `W = self.output_proj_weight`, `logit_chunk = mu_ @ w_c.transpose(-1, -2)`) reads only W, never the full-vocab mu/sigma tables (encode stays a per-token lookup). vfe3/run_artifacts.py:629-631: `fpt_estep = L * T * (2.0 * N * K + 2.0 * N * d_head * d_head)` counts only the belief E-step; `model_channel` (line 617-618, includes `cfg.s_e_step`) only bumps `active` (line 624-625), while the s-channel second E-step (vfe3/model/model.py:815 `self._refine_s(...)`, which runs a full `e_step(..., n_iter=cfg.n_e_steps, ...)` at model.py:631-633 once per forward) contributes zero FLOP terms. `git log --oneline f8ac3a0..HEAD -- vfe3/run_artifacts.py` is empty — no commit touched the file since the audit. ### Finding 6 — Shared prebuilt transport bypasses the truncated-backprop rebuild

**Audited:** medium | **Verdict:** STILL_PRESENT | **Current severity:** medium | **Confidence:** high

The bypass the audit described is unchanged in the current tree: e_step's truncated-backprop boundary rebuilds the hoisted flat transport from the detached phi only when prebuilt_transport is None, and the caller-shared shared_omega from model.py is built attached (grad-enabled) despite the code comment claiming it is already detached. Under share_refine_s_transport=True + e_steps_backprop_last>0 the last-k iterations therefore backprop through the shared transport to the phi/pos-phi encode tables -- exactly the gradient the internal-hoist rebuild (m10) severs -- so the perf-sharing toggle silently changes training gradients under truncation, with no config rejection of the combination. It is latent under the current working-tree configs (share is ON in both train_vfe3.py and ablation.py, truncation is 0), so activation needs one default-off toggle flip; forward values are unaffected either way.

**Evidence.** vfe3/inference/e_step.py:1005 "if _hoisted_omega is not None and prebuilt_transport is None:" -- the truncation-boundary rebuild from detached phi skips caller-supplied transports; the justification comment at e_step.py:1004 ("A caller-shared prebuilt_transport is already detached, so leave it") is false on grad-enabled paths: vfe3/model/model.py:799 builds "shared_omega = build_belief_transport(beliefs.phi, ...)" under "run = torch.no_grad() if e_step_gradient == 'detach' else nullcontext()" (model.py:775), and beliefs.phi is a live nn.Parameter lookup (vfe3/model/prior_bank.py:1009 "phi = pb.phi_embed[token_ids]", parameter at :190), so under 'unroll'/'straight_through' the shared transport is ATTACHED. It reaches the truncated e_step via model.py:847 -> stack.py:96 -> block.py:105 "prebuilt_transport=prebuilt_transport" alongside block.py:103 "e_steps_backprop_last=cfg.e_steps_backprop_last". No config guard: config.py:2201-2204 only checks e_steps_backprop_last >= 0. Current entry-point configs: share_refine_s_transport=True (train_vfe3.py:388, ablation.py:400) but e_steps_backprop_last=0 (train_vfe3.py:351, ablation.py:363).

**Recommended action.** At the e_step.py truncation boundary, drop the "and prebuilt_transport is None" condition and always rebuild the flat transport from the now-detached phi when _hoisted_omega is not None (values are identical at e_phi_lr==0, so the forward stays byte-identical and the m10 semantics become consistent across the internal-hoist and caller-shared routes); alternatively detach the caller-supplied transport's tensors at the boundary, and correct the false "already detached" comment.

### Finding 9 — Two-hop kernel and scalar free energy use different hop weights under saturation
