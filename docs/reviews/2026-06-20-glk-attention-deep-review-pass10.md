## Findings applied this pass

### F1 — RoPE interpolation hierarchy: directional wording backwards (attn ~1914, low)

The pass-9 rewrite recast the interpolation chain into the uniform transport-class form
`σ^{-2}Ω_ij^{-⊤} (GL(K)) ⊃ σ^{-2}Ω_ij^{-⊤} (const·SO(2)^{d_k/2}) ⊃ σ^{-2}Ω_ij^{-⊤} (const)`,
displayed largest-class-first (left to right the admissible class shrinks: the full group, then the rotation subclass, then the constant gauge). The following prose, however, read "each expression is the same logit kernel ... restricted to a progressively **larger** class of admissible transports," which is backwards relative to the displayed order. This is a genuine, if minor, directional error new to the pass-9 edit.

The transformer-ML and gauge-theory lenses disagreed on a second point — whether the set-membership predicate `Ω_ij ∈ const·SO(2)^{d_k/2}` is a category slip (the transformer lens read `const` as an arbitrary constant matrix, under which `{const} ⊄ const·SO(2)`; the gauge lens read `const` as the constant learned base gauge `Ω₀`, under which the nesting `{Ω₀} ⊂ {Ω₀·R} ⊂ GL(K)` is literal and the immediately-following prose, "Standard attention uses a constant, learned gauge ... RoPE augments this with a fixed SO(2) rotation sandwiched between the projections," fixes the reading). Adjudication: the underlying nesting is sound under the manuscript's own stated reading — `const·SO(2)^{d_k/2} ⊂ GL(K)` (det `c^K ≠ 0`, numerically confirmed), and standard attention is the zero-frequency (`R = I`) special case of RoPE — so the set-membership notation is defensible and was not overhauled. Only the unambiguous directional error was corrected.

Fix applied: "progressively larger" → "progressively smaller", with a parenthetical "(read left to right, from the full group to the constant gauge)" to remove the reading ambiguity that drove the lens disagreement.

One item adjudicated not-a-defect during the compile: `\href` at attn:2424 (the Code/Data-availability URL) is used without the manuscript loading `hyperref`. This is not a bug — the real `jmlr2e.sty` loads `hyperref` by default (`\@usehypertrue`; `\RequirePackage[colorlinks=false,allbordercolors={1 1 1}]{hyperref}`, verified at the JmlrOrg source), so `\href` is provided by the class. No `\usepackage{hyperref}` should be added to the manuscript (it would duplicate the class's load and risk a load-order clash).
