## Steelman of the opposing position
Total-vs-active is the standard sparse/MoE distinction (Shazeer, Fedus): a `V x n_gen` table that contributes one active row per token is genuine capacity that the per-token working set correctly excludes, so a flat active-param axis with falling CE is a real efficiency knob.

## My position (in service of red)
The "~12.06M essentially constant active working set" is a definitional consequence of the closed-form proxy at fixed `V,K`, not a measured efficiency. The proxy is a per-token READ COUNT that (a) excludes the `V x n_gen` bulk that actually grows and (b) is a parameter count, not a FLOP count, so it excludes the transport work that grows 64x. The proxy's own paired compute axis (`est_flops_analytic`) is flat only because the fixed decode term `2*V*K` swamps everything, and the empirical ground-truth compute (`wall_time_s`) is U-shaped with GL24 the SLOWEST run. Under the Shazeer/Fedus standard the flat-active-param claim is unsupported because it is not paired with a flat compute axis — here compute is not flat.

## Evidence
- `vfe3/run_artifacts.py:616-620` (REACHED: called at `run_artifacts.py:757`; `use_prior_bank=False` -> line 617-618 `active += V*K` fires; `model_channel=True` -> line 619-620 `active += 2*V*K` fires). Resolves to `active = 5*V*K + 2*K + n_gen`. With `V,K` fixed, the ONLY sweep-varying term is `+ n_gen`. `python`: base `5*50257*48 + 2*48 = 12,061,776`. CSV `active_params_per_token`: GL3=12,061,920, GL6=12,062,064, GL8=12,062,160, GL12=12,062,352, GL24=12,062,928 — each is exactly `12,061,776 + n_gen`. Total delta GL3->GL24 = +1,008 = +0.008 percent. The "flatness" is what the formula MUST return when `V,K` are held fixed; it is not an efficiency measurement.
- `vfe3/model/prior_bank.py:167` — `self.phi_embed = nn.Parameter(phi_scale * torch.randn(vocab_size, n_gen))`, shape `(V, n_gen)`; and `:682` — `phi = pb.phi_embed[token_ids]`, a token-indexed ROW lookup (REACHED: `_encode_per_token`, the `encode_mode='per_token'` path in this config). `V*n_gen` grows 7,237,008 -> 57,896,064; that is 99.74 percent of the total-param growth (`python`: `(phi24-phi3)/(70.16M-19.37M)`). The proxy at line 616 counts only the single looked-up belief row `2K + n_gen`, so it excludes 99.7 percent of what grows. Flat active-params is therefore compatible with 3.62x total-param growth by construction, not by efficiency.
- `vfe3/run_artifacts.py:625-627` with `d_head = K/n_blocks` at `:611` (REACHED). For `block_glk` at fixed K=48, `n_blocks=n_heads=48/g`, so `d_head=g`. The transport sub-term `2*N*d_head^2 = 2*N*g^2` grows 2,304 -> 147,456 (`python`: 64x, `24^2/3^2`). But `fpt_decode=2*V*K=4,824,672` is fixed and dominates `fpt_estep` (14,592 -> 159,744), so `est_flops_analytic` moves only 1.030x (CSV: 1.189e15 -> 1.225e15; `python`: `1224970076160000/1189297520640000 = 1.030`). The proxy's compute axis is flat as a DECODE-DOMINANCE artifact, hiding a 64x growth in the very E-step transport it purports to price.
- `wall_time_s` in `scaling_points.csv` (empirical ground truth per the docstring at `run_artifacts.py:604`) is U-shaped: seed-6 GL3=6168.6, GL8=4634.2 (min), GL24=11070.6. GL24 is the slowest run — 2.39x GL8 and 1.80x GL3. Real compute is not flat; the metric that is flat (active-params, and the decode-dominated analytic proxy) is precisely the one that does not see the transport blow-up.
- `vfe3/model/head_mixer.py:105-106` — `mixer_deltas` is one `nn.Parameter(torch.zeros(m,m))` per isotypic component, with `m = n_blocks = n_heads = 48/g`. So the head mixer SHRINKS 16x16 (GL3, 256 params) -> 2x2 (GL24, 4 params), zero-initialized — it moves opposite to the CE gain and cannot be the capacity driver.

## Newly-discovered context (for 01b_extended_evidence.md)
- The CSV carries BOTH `est_flops_6ND` and `est_flops_analytic`. `est_flops_6ND` grows 3.62x (it tracks `n_params`, i.e. counts `phi` as if fully active every step: 2.856e16 -> 1.034e17), while `est_flops_analytic` grows 1.03x (counts `phi` as decode-bound). The two proxies BRACKET the truth; neither is "the" compute, and the headline flatness depends entirely on which is chosen. The panel should treat the compute axis as unresolved, not settled at "flat."
- The active formula (line 616-620) does NOT count `output_proj_bias` (`decode_bias=True` here) nor the head mixer `m^2`; it counts `output_proj_weight = V*K` (line 618) as active because decode reads all V rows. So the "working set" is decode-read-count + one belief row — a legitimate number, but a READ count, orthogonal to both total capacity and to E-step FLOPs.
- `n_gen` and `n_blocks` are read from the live group object (`model.group.generators.shape[0]`, `len(model.group.irrep_dims)` at lines 609-610), so `d_head` is a REPRESENTATIVE block dim; for `block_glk` all blocks are equal, so `d_head=g` is exact, not an approximation.

## Falsification conditions
My implementation claim is wrong if: (1) `active_params_per_token` can vary by more than `+n_gen` when `V,K` are held fixed — it cannot, the closed form at line 616-620 forbids it; (2) `est_flops_analytic` were actually flat because the E-step genuinely costs the same across g — it is not, `fpt_estep` grows 10.9x (line 626) and is only masked by decode; (3) `wall_time_s` were flat or monotone-decreasing across the sweep — the CSV shows it is U-shaped with GL24 slowest. If a reviewer re-defines the compute axis as measured wall-time or a transport-inclusive FLOP count, the "flat working set" efficiency framing collapses; if they keep active-params, they must, per Shazeer/Fedus, pair it with a flat compute axis, which this sweep does not have.

## Confidence
HIGH — the formula, its runtime reachability, the CSV match to the integer, and the U-shaped wall-time are all verified against code and recorded artifacts. What would shift me: a transport-inclusive or wall-clock compute axis that is genuinely flat across GL3->GL24 (the CSV shows the opposite). The interpretive "publishable" verdict is the panel's, not mine.

## Canon (external source-of-truth for the metric's meaning)
- Fedus, Zoph, Shazeer 2021, "Switch Transformers" (arXiv:2101.03961): they "increase the parameter count while keeping the floating point operations (FLOPs) per example constant," on the thesis that "the parameter count, independent of total computation performed, is a separately important axis on which to scale," and comparisons are "FLOP-matched" (Table 9). The efficiency claim is licensed by holding a genuine compute axis fixed. `blocks_K48` holds a decode-dominated PROXY fixed while true compute (wall-time) rises.
- Shazeer et al. 2017, "Outrageously Large Neural Networks: The Sparsely-Gated MoE Layer" (arXiv:1701.06538): "Conditional computation, where parts of the network are active on a per-example basis," yields "dramatically increasing model capacity without a proportional increase in computation" — "greater than 1000x improvements in model capacity with only minor losses in computational efficiency." The distinction is legitimate, but its warrant is a near-flat compute cost, which the wall-time here (2.39x GL8->GL24) does not provide.
- Kaplan et al. 2020 (arXiv:2001.08361) / Hoffmann et al. 2022 Chinchilla (arXiv:2203.15556): the parameter-only power law is a fixed-data slice; a flat active-param axis with a 64x-growing transport cost and a 2x token-budget gap vs `grow_K_GL10` is exactly the confounded comparison these works warn against.
