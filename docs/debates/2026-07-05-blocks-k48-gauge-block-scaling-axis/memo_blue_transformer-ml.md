# Memo — debate-expert-transformer-ml — blue — round 1 (opening) — blocks-k48-gauge-block-scaling-axis

## Lens
Transformer architecture — token/input embedding tables, multi-head head-count-vs-head-width partition at fixed model width, the total-vs-active parameter distinction from sparse/conditional-computation models, and the head-mixer capacity account. (The softmax-form / `1/sqrt(d)` / RoPE machinery is out of scope for this scaling claim; nothing in it turns on Vaswani §3.2.1.)

## Steelman of the opposing position
The three co-varying knobs (total `n_params` up 3.62x, per-head dim `g` up, head count `H` down) mean the loss drop cannot be pinned to a "gauge-block scaling axis," and calling constant-active/growing-total "efficient" is the standard embedding-table tautology dressed as a scaling law over a compressed 3.62x x-range.

## Verdict
From this lens I firmly defend the mechanistic core: (1) `active/tok ~ constant` while total params grow is the textbook behavior of a token-indexed embedding table, not a metric artifact of `active_params_per_token`; (2) block enlargement at fixed `embed_dim=48` is a legitimate multi-head partition axis (head count traded for per-head dimension), genuinely distinct from the width axis that `grow_K_GL10` moves; (3) the head mixer is not the driver, since its capacity shrinks as the loss improves. I cannot certify the full compound clause "parameter-efficient scaling axis, publishable as-is": "parameter-efficient" is only meaningful paired with a compute axis (wall-time is U-shaped, GL24 is the slowest run, on half the token budget), and gauge-structure-vs-raw-capacity is untested without a non-gauge matched-size control. Recommend REMAND to the narrower true sub-claim.

## Argument

**Vector 1 — the constant-active/growing-total pattern is standard embedding-table behavior, not a metric trick.** `phi_embed` is a `(V, n_gen)` parameter (`prior_bank.py:167`) read by token-indexed row lookup, `phi = pb.phi_embed[token_ids]` returning `(B, N, n_gen)` — one row of width `n_gen` per token (`prior_bank.py:682`). This is structurally the input-embedding table every GPT-class model uses. Vaswani et al. 2017 §3.4: "we use learned embeddings to convert the input tokens and output tokens to vectors of dimension d_model." GPT-2 (Radford et al. 2019) realizes this as a `[50257 x 768]` token-embedding matrix used as a lookup table, one active row per token; the project's `vocab_size = 50257` is literally the GPT-2 byte-level BPE vocabulary (`01_evidence.md:11`). A table's total count `V x d` is large while its per-token working set is one row — so `active/tok` staying flat as `n_gen` grows (`+1008` total across the whole sweep = 0.008%, `01_evidence.md:50`) is the expected, well-understood property, not a proxy pathology. The sparse-model literature names this exact axis: Fedus et al. 2021 (Switch Transformer) "we investigate a fourth axis: increase the parameter count while keeping the floating point operations (FLOPs) per example constant... the parameter count, independent of total computation performed, is a separately important axis on which to scale," yielding "a sparsely-activated model — with an outrageous number of parameters — but a constant computational cost"; Shazeer et al. 2017 achieved ">1000x improvements in model capacity with only minor losses in computational efficiency" by activating a subset of parameters per example. The user's challenge ("vertical params/token? GL24 has MANY more parameters than GL3") is answered by exactly this distinction: the extra parameters live in the `V x n_gen` lookup bulk (99.7% of the growth, `01_evidence.md:49`), which is never touched per token except the single read row.

**Vector 2 — block enlargement is a recognized multi-head partition axis, distinct from width-scaling.** At fixed `embed_dim = 48`, the sweep sets `n_heads = 48/g` and per-head dim `d_head = g` (`01_evidence.md:13,30`) — moving `g` from 3 to 24 trades 16 heads of width 3 for 2 heads of width 24. That is precisely the head-count/head-width knob Vaswani et al. 2017 §3.2.2 defines: "linearly project the queries, keys and values h times ... to d_k, d_k and d_v dimensions ... For each of these we use d_k = d_v = d_model/h = 64. Due to the reduced dimension of each head, the total computational cost is similar to that of single-head attention with full dimensionality." `grow_K_GL10` moves the orthogonal knob — it grows `d_model = 10 * n_heads` at fixed block, adding heads at fixed per-head dim. Fixing `d_model` and repartitioning it (this sweep) versus growing `d_model` by adding heads (`grow_K_GL10`) are two different coordinates in the Vaswani parameterization, so "distinct/complementary axis" has real architectural content — it is not one knob relabeled. That loss falls strictly monotonically toward fewer/wider heads at fixed `d_model` (PPL 124.6 -> 92.2, seed-robust) is a legitimate design-axis finding about the head-count/head-width tradeoff.

**Vector 3 — the head mixer is not the driver.** The learned head mixer is a per-block map on `H = n_heads` heads, so it shrinks from `16x16` (GL3) to `2x2` (GL24) as blocks enlarge (`01_evidence.md:34`). Its capacity moves opposite to the loss improvement: loss improves as the mixer gets smaller. A "the toggle smuggled in the gain via head-mixer capacity" attack is therefore ruled out on monotonicity grounds — the gauge-block structure, not the mixer, tracks the improvement.

## Concessions / limits
- The embedding-table analogy is not exact. In a plain table the read row is the representation used directly; here `phi` parameterizes a gauge element that then enters a KL/transport computation. The Switch/Shazeer active-vs-total framing still holds (the read row is active, the `V x n_gen` bulk is not), but the single active row does structured compute, not lookup-then-done. State this precisely rather than claiming a pure MoE.
- "Parameter-efficient" needs the compute axis Fedus explicitly requires. The analytic FLOP proxy is decode-dominated and nearly flat, but empirical wall-time is U-shaped and GL24 is the slowest run (`01_evidence.md:52`), on 245.76M tokens versus `grow_K_GL10`'s 491.52M (`01_evidence.md:19,54`). Half the token budget is a first-order confound for any absolute-loss comparison.
- Vector 3 rules out the head mixer but not all learned co-moving capacity. Under `use_prior_bank=False`, decode is the learned linear projection `logits = mu @ W^T` with a `(V, K)` `output_proj_weight` (`prior_bank.py:167-178`); it composes as `logits = (W M) mu`. The wider `phi` row (`n_gen = 48 g`) plus `W` are learned capacity that grows with the sweep. Whether the gain is gauge-block structure per se or raw added table capacity is untested: no non-gauge matched-size `V x m` control was trained (`01_evidence.md:80`).
- "Scaling axis" with a headline exponent is not supported by 5 points over a 3.62x `n_params` range; the fitted exponent is axis-dependent (0.929 vs `n_params`, 0.181 vs `n_gen`, degenerate vs FLOPs, `01_evidence.md:61`). Defend the *direction* as a real architectural finding; do not defend a reportable efficiency exponent.

## Falsification conditions
This position is wrong on transformer-architecture grounds if: (a) a runtime trace shows more than one `phi_embed` row participating per token, breaking the embedding-table account (`prior_bank.py:682` says otherwise); (b) a non-gauge learned `V x m` table of matched size, or a matched-param dense head-mix, recovers the same PPL curve — that would show the gain is raw table capacity, not gauge-block structure, collapsing "distinct axis"; (c) a matched 491.52M-token blocks_K48 run erases the GL3->GL24 gap, exposing the drop as the token-budget confound; (d) freezing/zeroing the learned decode `W` and head mixer at fixed `g` removes the monotonic improvement, which would move the credit off the gauge block.

## Newly-discovered canon (for 01b_extended_evidence.md)
- Vaswani et al. 2017, "Attention Is All You Need," NeurIPS, §3.2.2 and §3.4 (https://arxiv.org/abs/1706.03762). §3.2.2 fixes `d_model` and splits it into `h` heads of dim `d_model/h` at "similar" total cost — the exact head-count/head-width partition this sweep moves. §3.4 defines the learned input-embedding table (one active row per token).
- Radford et al. 2019, "Language Models are Unsupervised Multitask Learners" (GPT-2). Token-embedding matrix `[50257 x 768]` as a lookup table, one active row per token; `V = 50257` byte-level BPE is exactly the project's `vocab_size`, anchoring the `phi_embed` embedding-table analogy.
- Fedus, Zoph, Shazeer 2021, "Switch Transformers," arXiv:2101.03961. "Increase the parameter count while keeping the FLOPs per example constant ... a separately important axis on which to scale"; "a sparsely-activated model — with an outrageous number of parameters — but a constant computational cost." The canonical statement that total-vs-active is a real axis, but paired with a FLOP axis.
- Shazeer et al. 2017, "Outrageously Large Neural Networks: The Sparsely-Gated Mixture-of-Experts Layer," arXiv:1701.06538. ">1000x improvements in model capacity with only minor losses in computational efficiency" via per-example activation of a parameter subset — the origin of the conditional-computation active-vs-total distinction.

## Confidence
MEDIUM-HIGH on the mechanistic sub-clauses (embedding-table pattern, multi-head partition axis, head mixer not the driver); LOW on the full "parameter-efficient / publishable scaling axis" clause. A matched-token-budget blocks_K48 run and a non-gauge matched-size table control would settle whether the improvement is gauge-structure or raw capacity, and whether "efficient" survives the compute axis.
