# Memo — debate-expert-ml-engineer — BLUE — rebuttal — blocks-k48-gauge-block-scaling-axis

## Lens
General ML / optimization — scaling-law canon (Kaplan, Hoffmann), conditional-computation total-vs-active capacity (Shazeer, Fedus), parameter-count conventions, held-D loss slices, and the distinction between a compute axis and a metric proxy.

## Steelman of the opposing position
An active-parameter number becomes an *efficiency* claim only when the total-vs-active split is paired with a flat compute axis (Fedus, Zoph & Shazeer's "FLOPs per example constant"); here the one empirical compute measure, wall-time, is U-shaped with the best-loss point (GL24) the slowest run, so the flat 12.06M working set is a definitional proxy, not a measured efficiency.

## My position (in service of BLUE)
Within `blocks_K48` at a single fixed budget D = 245.76M tokens, the strictly monotone, three-seed CE decrease (PPL 124.6 -> 92.2) is a legitimate held-D, total-vs-active *capacity* observation whose bookkeeping — excluding the `(V, n_gen)` `phi_embed` table from the per-token working set — is doubly literature-sanctioned (Kaplan's non-embedding N; Shazeer/Fedus total-vs-active), not a metric trick. What is unearned, and conceded outright, is the compute-*efficiency* reading, the power-law *exponent*, and the cross-sweep "complementary to width-scaling" comparison.

## Reclaiming the canon Red borrowed

**Kaplan sanctions the phi_embed exclusion; it is the prescription, not a dodge.** Red frames the flat working set as "an algebraic identity, not an efficiency measurement." But the convention of dropping the vocabulary table from the parameter count that indexes a scaling curve is exactly what Kaplan et al. legislate. Section 2.1: "We use N to denote the model size, which we define as the number of *non-embedding* parameters." Figure 6: "When we include embedding parameters, performance appears to depend strongly on the number of layers in addition to the number of parameters ... When we exclude embedding parameters, the performance of models with different depths converge to a single trend." `phi_embed` is a `(V, n_gen)` table (prior_bank.py:167) read one row per token (prior_bank.py:682) — structurally an embedding table, one active row per token, growth uncorrelated with per-token compute. Counting its `V*n_gen` bulk *out* of the working set is not sleight of hand; it is the Kaplan convention applied to a Kaplan-shaped object. Red's own steelman already concedes this: "counting only active parameters is the standard and defensible bookkeeping." "Definitional" is not an indictment of a convention that matches canon — Kaplan's own N is equally definitional.

**Hoffmann's law predicts the sign of the within-sweep effect and certifies it as capacity-driven.** Red weaponizes `L(N,D) = E + A/N^a + B/D^b` to void cross-sweep comparisons. Grant that (below). But turn the same law on the *within*-sweep slice: all five `blocks_K48` cells share one identical D = 245.76M (01_evidence.md:38, same `data_sha256`), so the `B/D^b` floor term is byte-identical across the sweep, and the entire CE gap reduces to the `A/N^a` capacity term. Total N rises 3.62x while tokens-per-parameter *falls* (12.7 at GL3 to 3.5 at GL24) — GL24 is more data-starved per parameter, yet has the lowest loss, which is precisely the held-D N-scaling Hoffmann's law entails. The monotone improvement is not an artifact of more training (D is fixed); it is the loss surface's held-D slice. Kaplan and Hoffmann *describe* such slices; they forbid reading an infinite-data exponent off one, not the slice's existence.

## Conceded honestly (required)
- **The exponent is not reportable.** Axis-dependent 0.929 vs `n_params` / 0.181 vs `n_gen`, CI [0.07, 1.73], dynamic range under one decade against Stumpf & Porter's two-decade floor. Blue does not defend this as a power law; the 3.62x N-range is compressed because the fixed vocab tables dominate.
- **The cross-sweep comparison is a Chinchilla D-slice confound.** `blocks_K48` (245.76M) versus `grow_K_GL10` (491.52M) is a 2x D gap; under `L(N,D)` the "GL24 sits on the grow frontier / complementary to width-scaling" reading is non-identified. Conceded.
- **The efficiency conjunct fails on real compute.** Shazeer/Fedus license total-vs-active *only* against a flat compute axis (Fedus: "increase the parameter count while keeping the floating point operations per example constant ... a separately important axis"). Wall-time is U-shaped, GL24 slowest at 2.39x the GL8 minimum. On wall-clock the efficiency reverses; the "parameter-efficient" adjective is unproven as a *compute* claim. What survives is a *capacity* axis (Shazeer: capacity "without a proportional increase in computation"), not yet an efficiency axis.

The fork Blue must own: you may have the flat working set (exclude the table, Kaplan/Shazeer) or an N-scaling explanation of the drop (include the table, Hoffmann total-N), but the drop lives in the *excluded* table, so this is a total-vs-active *sparse-capacity* story, not a dense-compute-scaling law. That is coherent and honest — it just is not the compound claim.

## The winning experiment (Blue's own falsification condition)
One artifact converts Red's REMAND into a Blue win: a **matched 491.52M-token `blocks_K48` run** (removes the D-confound; puts blocks and grow on one D-slice) **plus a non-gauge matched-parameter `V x m` table control** at the same head geometry (isolates structured gauge capacity from raw table capacity). If the GL3->GL24 CE improvement survives at matched D *and* the plain table fails to reproduce the GL24 gain, the axis is earned. Blue stakes the claim on exactly this.

## Falsification conditions (this position is wrong on ML grounds if)
- At matched D = 491.52M the GL3->GL24 CE improvement collapses — then it was a small-N / D-slice artifact, not a capacity effect.
- A non-gauge multi-head baseline swept 16 heads x d_k=3 -> 2 heads x d_k=24 at d_model=48, with no `phi_embed` growth, improves monotonically on its own — then the gain is head-geometry recovery from the pathological d_k=3 regime, not held-D capacity scaling, and the capacity frame is the wrong one.
