# Memo — debate-expert-transformer-ml — red — rebuttal — blocks-k48-gauge-block-scaling-axis

## Lens
Transformer architecture — token-embedding table (Vaswani §3.4, GPT-2), the multi-head partition `d_model = h·d_k` (Vaswani §3.2.2), head-count redundancy, and embedding-vs-non-embedding parameter accounting (Kaplan et al.).

## Steelman of the opposing position
`phi_embed (V, n_gen)` read one row per token is the canonical token-embedding table, so a flat `active_params_per_token` at growing `n_gen` is the expected, well-understood property of a lookup table (Vaswani §3.4, GPT-2's `[50257 × 768]`), not a metric pathology — which legitimizes reporting the low per-token working set.

## My position (in service of red)
The embedding-table identity Blue invokes to legitimize the metric is the same identity that converts the GL3→GL24 loss gain into a capacity/architecture confound rather than a gauge-structure mechanism. Two canonical non-gauge knobs co-vary with the block size and are each sufficient to explain the descent: (i) the per-token learned `phi` row width grows 8× (`n_gen` 144→1152), which is embedding-table capacity that a plain non-gauge `(V, m=n_gen)` table read one row per token would supply identically; and (ii) the fixed `d_model=48` head partition is slid from 16 narrow heads (`d_k=3`) to 2 wide heads (`d_k=24`), a Vaswani §3.2.2 multi-head-partition move on a redundancy-laden axis. Blue's own analogy therefore predicts a non-gauge table reproduces the gain — so "enlarging the gauge block is a distinct/efficient scaling axis" is not established. I concede `phi_embed` is genuinely embedding-table-like (one row per token) and that the head mixer shrinks (16×16→2×2) and is not the driver; neither concession rescues the axis claim.

## Evidence
- **Vaswani et al. 2017, "Attention Is All You Need," NeurIPS, §3.2.2** — verbatim: *"d_k = d_v = d_model/h = 64 … Due to the reduced dimension of each head, the total computational cost is similar to that of single-head attention with full dimensionality,"* and *"Multi-head attention allows the model to jointly attend to information from different representation subspaces at different positions."* Mapping the sweep: `n_heads = 48/g`, `d_k = g`, `d_model = 48` fixed, so GL3→GL24 is exactly `(h=16, d_k=3) → (h=2, d_k=24)` — one traversal of the multi-head partition of a single fixed `d_model`. Vaswani's own configuration uses `d_k=64`; the GL3 endpoint runs 16 heads at `d_k=3`, a severe per-head dimension starvation, and the improvement is consistent with relieving that partition bottleneck by widening heads, independent of any gauge story.
- **Michel, Levy, Neubig 2019, "Are Sixteen Heads Really Better than One?" NeurIPS, arXiv:1905.10650** — verbatim: *"a large percentage of attention heads can be removed at test time without significantly impacting performance … some layers can even be reduced to a single head."* Head count is a redundancy-laden knob, so moving 16→2 heads is not a clean capacity variable that can be credited to "richer gauge block."
- **Voita, Talbot, Moiseev, Sennrich, Titov 2019, "Analyzing Multi-Head Self-Attention," ACL, arXiv:1905.09418** — verbatim: *"Specialized heads are last to be pruned. Our novel pruning method removes the vast majority of heads without seriously affecting performance. For example, on the English-Russian WMT dataset, pruning 38 out of 48 encoder heads results in a drop of only 0.15 BLEU."* Quantifies that the head-count face of `d_model = h·d_k` carries little independent signal — the sweep is moving a knob the literature has shown to be largely slack.
- **Kaplan et al. 2020, "Scaling Laws for Neural Language Models," arXiv:2001.08361, §3.1** — verbatim: *"Transformer performance depends very weakly on the shape parameters nlayer, nheads, and dff when we hold the total non-embedding parameter count N fixed,"* with *"N — the number of model parameters, excluding all vocabulary and positional embeddings."* Two-pronged against Blue's Pillar 2: (a) `nheads` is a weak knob at fixed non-embedding `N`, so the 16→2 head co-variation is not the mechanism, which leaves the growing table as the driver; (b) Kaplan brackets embedding parameters OUT of the clean scaling axis because they scale differently — and Blue concedes `phi_embed` is exactly such an embedding table, so the 124.6→92.2 gain is being bought in the excluded-from-`N` embedding bucket, i.e., raw per-token table capacity, precisely what a non-gauge `(V, m)` table delivers.
- **Vaswani §3.4 / Radford et al. 2019 (GPT-2), embedding table** (conceded, but re-weighted): the embedding table gives each token a learned vector of width `d_model`, read one row per token — the canonical per-token capacity axis. Here `n_gen` is decoupled from `embed_dim=48` and grown 8× (144→1152); Blue's "+0.008%" is small only against the fixed ~12.06M decode denominator it is measured against. The token-distinguishing learned capacity — the very thing an embedding table exists to provide — grows 8×, and its being swamped by the decode term does not make it "free"; it makes `active_params_per_token` insensitive to the capacity that is actually moving.

## Falsification conditions
This red position is wrong on transformer-architecture grounds if any of the following holds:
1. **Plain-table control reproduces the gain but is claimed to be gauge — reversed.** If a trained non-gauge `(V, m=n_gen)` learned table read one row per token, at matched added params and matched 245.76M tokens, FAILS to reproduce the 124.6→92.2 curve, then the gain requires gauge structure and my "any table would do it" claim falls.
2. **Head-partition-isolated control.** If a run that holds `n_heads` (equivalently `d_k`) fixed while varying only the per-block `gl(g)` algebra dimension recovers the full descent, the multi-head-partition confound is not load-bearing and my leg (ii) fails.
3. **Kaplan shape-invariance does not transfer at tiny `d_k`.** If a fixed-`d_k`, varied-gauge run still moves loss substantially, then `d_k=3` starvation is not the operative bottleneck and the head-width reading is refuted — leaving only the table-capacity leg (which still confounds Blue, but my two-pronged framing weakens to one prong).

## Confidence
HIGH — that Blue's embedding-table analogy makes the loss gain a capacity confound (a matched non-gauge table control is missing, and Kaplan explicitly excludes embedding growth from the clean scaling axis). MEDIUM on the precise split of credit between table-capacity and head-partition. A trained non-gauge `(V, n_gen)` table that fails to reproduce the curve at matched params/tokens would shift me toward Blue.
