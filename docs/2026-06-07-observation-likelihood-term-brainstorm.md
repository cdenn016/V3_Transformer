# What an observation-likelihood term would mean in the VFE3 next-token language model, and whether to thread or remove the dead `log_likelihood` argument

## 1. Why the question is subtle

The user asks what the canonical free energy's observation term `-sum_i E_{q_i}[log p(o_i|k_i)]` could mean when the model performs next-token prediction and it is not clear what the observations are. The difficulty is structural rather than terminological: in this architecture the two roles a standard active-inference observation would occupy are already taken by other mechanisms, and they are taken in two different loops. The present token sets the prior. The `encode` method returns the initial belief equal to the prior, `q_i(0) = p_i`, looked up from the input token (`vfe3/model/prior_bank.py:178`, "Look up the per-token Gaussian prior as the initial belief (q = p)"), and the same prior table feeds the self-coupling `alpha*KL(q_i||p_i)`, the encode initialization, and the decode readout. The belief is grounded to the present symbol before any inference runs. The next token, by contrast, drives the loss discriminatively in the outer loop: `decode` produces logits `l_v = -KL(q_i||pi_v)/tau_eff`, then `F.cross_entropy(logits, targets)` is computed in `model.py` only after the belief stack returns. The inner E-step that minimizes the VFE over `(mu, Sigma, phi)` is provably blind to the targets; `free_energy_value` and the whole belief-descent chain take no target channel, a fact corroborated independently by the project's own observation log (16842, 16845).

Because the data has been split into a prior (the present token) and a discriminative loss (the next token), the canonical free energy's observation term has no place to land, and this is the structural reason the `log_likelihood` argument is unreachable. It is declared at `vfe3/free_energy.py:268` and subtracted at `:302-303` as `F = F - log_likelihood.sum()`, yet no production caller supplies it; a codebase-wide check confirms zero call sites pass the argument (observation 16846).

A second subtlety sharpens this further: `free_energy()` is not even on the default belief-descent path. Belief means and variances descend via the analytic kernel `belief_gradients -> get_kernel` (`vfe3/gradients/kernels.py:179-186` gate the default `use_kernel` branch, and `:209` calls `get_kernel`, which never invokes `free_energy()`). The functional `free_energy()` is reached only when `use_kernel=False` routes through the autograd oracle (`vfe3/gradients/oracle.py:110`, which builds `sd`, `energy`, and `alpha` locally at `:105-109` and calls `free_energy()` directly), or through the discarded `no_grad` diagnostic logger `_f_diag` (`vfe3/inference/e_step.py:444`). Threading `log_likelihood` into `free_energy()` alone would therefore be inert on every default run, exhibiting the same half-wiring pathology the argument already shows. Any genuine observation pull must be injected into the kernel and the oracle, not into `free_energy()` in isolation.

A third reframing is worth stating at the outset because it answers the question at the level of intent. In Wheeler's participatory sense the model is not missing its observation. The act of eliciting `l_v = -KL(q_i||pi_v)/tau` and scoring it against the realized next token through cross-entropy is itself a measurement: each vocabulary token poses a yes/no question, the softmax partition enforces exactly-one-yes, and the realized token is the elicited bit. On this reading the dead `log_likelihood` argument is the generative cousin of an observation the model already realizes discriminatively in the outer loop, and the codebase says so itself. The diagnostic at `vfe3/metrics.py:634` literally names the data term, `data_term: # -E_q[log p(o|x)] = val_ce in nats`, and folds it into the displayed free-energy total at `:665-666`. The diagnostic stack already identifies cross-entropy as the data term, which is both the strongest in-code support for the observation-is-cross-entropy reading and a genuine double-count trap.

## 2. The cross-entropy versus observation identity, stated correctly

For the pure KL decode, `-KL(q_i||pi_v) = H(q_i) + E_{q_i}[log pi_v]`, so the logit is `l_v = (H(q_i) + E_{q_i}[log pi_v])/tau`, and the per-position training cross-entropy satisfies

```
tau * CE_t  =  -E_{q_i}[log pi_t]  +  tau * logsumexp_v( E_{q_i}[log pi_v] / tau ).
```

The belief entropy `H(q_i)` is independent of the vocabulary index: it factors out of the logsumexp and cancels exactly against `-l_t`. The only gap between the manuscript's per-position observation term and the training cross-entropy is therefore the vocabulary log-partition `tau*logsumexp_v(...)`, and that term is not constant in `q_i`, so it does not absorb away. The first term `-E_{q_i}[log pi_t]` is the manuscript's observation term (PIFB:1949, with the Gaussian appearance model of PIFB:1958-1960) under the appearance model `p(o=t|k) = pi_t(k)`, in which each vocabulary token is a Gaussian in latent space. The training cross-entropy is thus the discriminative, vocabulary-normalized cousin of the generative additive observation term, and the choice between an additive generative pull and a vocabulary-competitive normalization is a real design fork.

One precision caveat belongs here. PIFB:1958 writes the likelihood as a continuous Gaussian `N(o_i; theta, Sigma_o)` over a continuous observation `o`; the categorical reading `p(o=t|k) = pi_t` is this document's own interpretive bridge from that continuous Gaussian likelihood to the discrete vocabulary, not the manuscript's literal form. The bridge is defensible because each vocabulary entry already carries a latent-space mean and variance, but it should be flagged as a reinterpretation, since the continuous-versus-categorical character of `o` is exactly what separates the taxonomy's continuous-embedding rows from its present-token row below.

The first-draft attribution of this gap requires one correction. An earlier expert reading claimed that Fact 1 mislabels the gap and that there are two stacked gaps, the log-partition and `H(q_i)`. This is incorrect for the training loss, and the error is resolvable by algebra rather than by vote: `H(q_i)/tau` is constant in the vocabulary index and cancels completely in the softmax cross-entropy, leaving the log-partition as the sole gap. The expert conflated two distinct identities. The softmax-over-vocabulary training loss versus the observation term has a gap equal to the log-partition, with `H(q)` cancelling; the single-agent comparison `-E_q[log q_e]` versus `KL(q||q_e)` has a gap equal to `H(q)` (PIFB:1497). Both identities are true, but the second was applied to the first's object. The claim is relocated, not discarded: the expert's `H(q)` characterizes the environmental-agent-coupling versus likelihood gap, which is the engine of the environmental-agent interpretation in the taxonomy below. It is the difference between routing an observation through a `beta*KL` coupling block and routing it through a true likelihood, which is precisely the covariance-sector object that distinguishes the two and the reason the manuscript's exact-ELBO fix takes the cross-entropy form `-E_q[log q_e] = KL + H(q)` (PIFB:1497-1500).

Differentiating the identity in the mean sharpens its content. We have `d(tau*CE)/dmu = Lambda_t(mu - mu_t) - E_{v~p_model}[Lambda_v(mu - mu_v)]`. The first term is exactly the generative observation gradient `Lambda_o(mu_i - o_i)` (PIFB:1960) with `o = mu_t`; the second is the log-partition gradient, the contrastive negative phase. The generative observation term is therefore literally the cross-entropy gradient with the negative phase deleted. This is the optimizer-level statement of the fork: the additive generative pull is not self-limiting, vanishing only when `mu = o`, whereas softmax cross-entropy's gradient vanishes as the prediction becomes confident and correct.

The covariance sector deserves its own statement rather than a parenthetical, because the manuscript's Gaussian likelihood at PIFB:1960 carries two pieces, the mean-pull `(1/2)(o-mu)^T Lambda_o (o-mu)` and the covariance shrinkage `(1/2)tr(Lambda_o Sigma_i)`, and the brief explicitly asks about the latter. The covariance shrinkage is an unconditional, floor-free precision injection: its gradient is `dF/dSigma = +(1/2)Lambda_o`, monotone and never vanishing. The cross-entropy covariance gradient, by contrast, is `dCE/dSigma_q = (1/2tau)(Lambda_t - E_beta[Lambda_v])`, a contrast that vanishes at confident-correct prediction. This is the real generative-versus-discriminative fork in the covariance sector, distinct from the well-covered mean-sector fork, and it is exactly why a naive observation term risks driving `Sigma -> 0` overconfidence collapse. That collapse is not hypothetical for this codebase: the `eigh` backward is known to produce NaN under exact eigenvalue degeneracy (observation 16360), and an unfloored precision injection drives `Sigma` toward the degeneracy that triggers it. What the model already does is shape `Sigma` through cross-entropy; what a generative term would add is unconditional precision injection with no floor.

The first-draft single row for teacher or distillation embeddings is split here into a teacher-network row and a teacher-free empirical-soft-distribution row, because the latter is buildable today without any external model and the former is not. One correction to the present-token row is a consensus sharpening rather than an expert split. The first-draft brief's framing that "two Gaussians on the same point just multiply, so the term is redundant" is precise on the mean but understated on the covariance. The precision sector is not a no-op, since `Lambda_p -> Lambda_p + Lambda_o` sharpens `Sigma`, but it is foldable into `alpha` and carries no information distinct from the prior. The decision-relevant column, distinct information versus the prior, is therefore unanimously no, and row (a) is a reparametrization of `alpha` rather than a new term.

## 4. The shadow-prior coupling

The shadow-prior fact is correct and is the load-bearing constraint of the entire analysis. PIFB:1233 states verbatim that the per-token PriorBank `p_i` is the frozen-slow-subsystem special case of the cross-scale shadow prior `p_i = Omega_{i,I}[q_I^{(s+1)}]` (PIFB:547). In the general hierarchy the prior is supplied top-down by a coarser-scale belief. The reason there is no room for an observation in the present design is precisely that the present token is doing double duty as prior and as data.

Free the prior to be top-down, the shadow, and the present token is liberated to act as the bottom-up observation, recovering the textbook active-inference decomposition in which the prior is the top-down prediction, the observation is the bottom-up data, and the prediction error `mu_i - o_i` drives the belief. Only then does the present token carry information distinct from the prior, and only then is `-E_q[log p(o|x)]` simultaneously non-vacuous, EM-clean, and manuscript-faithful, which is row (k) of the taxonomy. Every lens converges on this conclusion: the variational, information-geometric, gauge, geometry, philosophy, and systems analyses all land on the observation term being gated on the shadow prior. The observation term is entangled with and gated on the shadow-prior work; it is not a standalone wiring fix.

The gauge lens adds the cleanest statement of what becomes possible. In the frozen special case the vocabulary Gaussians carry an implicit identity frame, since `pi_v` has a mean and variance but no `phi_v` slot, so decode is a gauge-fixing soldering of the transported belief against an untransported reference. Once the prior is a transported top-down shadow, the present token can enter as a gauge-covariant observation in the sensor frame, `-KL(q_i || Omega(phi_i, phi_v^obs) . pi_v)`, the external source that pins the otherwise-free global-diagonal gauge. Observation, in the gauge reading, is what fixes the gauge.

## 5. Recommendation: keep a documented stub, gated on the shadow prior

The committed position, stated without hedge, is to keep `log_likelihood` as a one-line documented stub explicitly gated on the shadow-prior roadmap (#1/#2), to not thread it into `free_energy()` now, and to not silently delete it.

The apparent split between a lone "remove" verdict and a "keep" majority is narrower than it appears. The remove-leaning verdict was in fact "remove or reduce to a one-line documented stub gated on the shadow-prior work," which coincides with the keep position; the gated stub subsumes rather than refutes a standalone remove-position. The gated stub is chosen over a bare delete because CLAUDE.md mandates that a theoretically pure path always exist under an appropriate toggle, which a bare delete forecloses and a documented stub preserves. All readings agree on the two operative claims: do not thread the term standalone now, and do not silently delete it.

The reasoning, taken one reachable reading at a time, is that in today's next-token language model every threadable observation is either leakage (the next token into the E-step, row (b), which breaks EM separation and is non-causal), double-counting (the next token in the outer loop, row (c), where cross-entropy already lives and `metrics.py:634` already books it as `data_term`), or a re-skinned `alpha` (the present token, row (a), redundant with the prior). The only genuinely useful generative reading needs an input channel that does not yet exist (rows (d), (h-teacher), (h-empirical), and (k)). A standalone E-step thread is therefore dominated: it is redundant if EM-safe and EM-breaking if informative.

Several conditions would flip the decision, and they should be stated rather than pre-decided. The decision flips to building the term now if a genuinely distinct channel arrives without waiting for the shadow prior: either an external or multimodal embedding (rows (d) and (h-teacher)), a real exogenous K-vector distinct from both prior and target, or a masked or denoising objective (row (f)), where masking directly breaks the `o = prior` identity so that a present-position observation becomes non-vacuous, causal, and EM-clean. A teacher-free soft-distribution target (row (h-empirical)) is the lowest-friction of these, since it requires no external model. The decision flips to outright removal if the shadow-prior roadmap is abandoned, because then there is nothing left to gate on, no external channel is planned, and the argument is pure dead weight inviting the next reader to mis-thread it. The decision stays a gated stub, which is the current recommendation, as long as the shadow prior remains a live roadmap item.

When the term is eventually built, the observation pull must be injected into the analytic kernel (`vfe3/gradients/kernels.py`, the `get_kernel` path at `:209`, for example as a self-like term in the gradient assembly) and into the autograd oracle (`vfe3/gradients/oracle.py:110`), not into `free_energy()` alone, which is off the default descent path and would silently no-op. The `log_likelihood` argument in `free_energy()` is correctly placed only as the oracle-path mirror. A point of citation precision matters here, because the codebase has two parallel free-energy assemblers and conflating them invites a mis-thread. The diagnostic wrapper `free_energy_value` (`vfe3/inference/e_step.py:138`) is called only by the `_f_diag` logger (`e_step.py:444`, under `no_grad`) and the visualization trajectory extractor (`vfe3/viz/extract.py:222,228`); the oracle (`oracle.py:110`) calls `free_energy()` directly using locally built `sd`, `energy`, and `alpha` (`oracle.py:105-109`) and does not route through `free_energy_value`. A bare reference to the E-step value function therefore risks the false impression that the oracle's descent flows through the diagnostic wrapper, which it does not. The concrete stub action is to add one comment line at `free_energy.py:268` stating that `log_likelihood` is the oracle-path mirror of the manuscript observation term, that it is gated on the shadow-prior work (PIFB:1233), and that a live observation must also enter `vfe3/gradients/kernels.py` and `vfe3/gradients/oracle.py` so that a future contributor does not "complete the canonical functional" by adding `log_likelihood` to `free_energy()` plus a single caller and believing it live. The blast radius is one comment, with no code change and no behavior change.

## 6. Genuine disagreements among the analyses, adjudicated

Four disagreements warrant explicit adjudication. The first concerns where the cross-entropy gap lives. It is resolved by the algebra of Section 2: the belief entropy cancels in the softmax cross-entropy, so the sole training-loss gap is the vocabulary log-partition, and the entropy identity belongs instead to the environmental-agent-coupling-versus-likelihood comparison. The second concerns whether the present-token redundancy is a split or a consensus, and it is a consensus: mean-redundant, precision-foldable into `alpha`, distinct information no, presented as a correction to the present-token reading rather than as a fork. The third concerns thread versus remove, which is narrower than it appears, since the remove-leaning verdict explicitly offered a documented gated stub, coinciding with the majority; it is adjudicated to the committed gated-stub position, with the "free_energy() is off the descent path" finding folded in as a constraint on how to thread later rather than as a competing verdict. This is the one place where the code is the decisive tiebreaker, and it is verified: the default `belief_gradients -> get_kernel` path (`kernels.py:179-209`) bypasses `free_energy()`, which is reached only through the oracle (`oracle.py:110`) or the discarded diagnostic logger (`e_step.py:444`). The fourth concerns whether the model already performs observation. Both answers are correct at different levels: participatorily, decode-and-score is the measurement and `metrics.py:634` names it the data term; generatively, the manuscript's additive `-E_q[log p(o|k)]` is genuinely absent and differs from cross-entropy by the non-constant partition. The honest framing carries both, namely that the observation is present discriminatively and absent generatively, and that the generative form becomes reachable only through the shadow prior.

The primary-source anchors for the analysis are as follows. In code: `vfe3/free_energy.py:268,302-303` (the dead argument); `vfe3/inference/e_step.py:138-226` (the target-blind diagnostic F-value wrapper, distinct from the oracle's direct call); `vfe3/gradients/kernels.py:179-209` (the default kernel path that bypasses `free_energy()`); `vfe3/gradients/oracle.py:105-110` (the oracle as the only `free_energy()` descent route, building its inputs locally); `vfe3/model/prior_bank.py:178` (the present token as prior and initial belief, `q = p`); and `vfe3/metrics.py:634,665-666` (`data_term := val_ce` folded into the displayed F). In the manuscript: PIFB:1949 (the observation term); PIFB:1958-1960 (the Gaussian likelihood, with mean-pull `Lambda_o(mu_i-o_i)` and covariance shrinkage `(1/2)tr(Lambda_o Sigma_i)`); PIFB:1497-1500 (`-E_q[log q_e] = KL + H(q)`, the environmental agent); PIFB:1233 and 547 (the PriorBank as the frozen special case of the top-down shadow prior); PIFB:1227 and 1244 (gauge transport and soldering); and PIFB:4692 (the leaf observation term). The project's own observation log independently corroborates the central claims: 16846 (zero call sites pass `log_likelihood`), 16843 (the `data_term` double-count), 16842 and 16845 (the observation term absent from the gradient and E-step paths), and 16360 (the `eigh` degeneracy NaN hazard that underwrites the floor-free covariance-collapse risk).

## 7. Open questions for the user

The recommendation above is committed, but its dependencies hinge on facts only the user can supply, and the answers determine which branch the decision takes. First, is the shadow-prior roadmap (#1/#2) still a live item? If it is, the gated stub is correct; if it has been abandoned, the argument should be removed outright, since there would be nothing left to gate on. Second, is any external or multimodal embedding channel planned, that is, a real exogenous K-vector distinct from both the present token and the next token? An affirmative answer flips the decision toward building the generative observation term now via rows (d) or (h-teacher). Third, is a masked or denoising objective something to pursue as a separate research track? Masking is the one regime in which a present-position observation becomes non-vacuous without the shadow prior (row (f)), so this is a standalone build path independent of #1/#2. Fourth, is a soft-target source available or desired, whether an external teacher network (row (h-teacher)) or a teacher-free empirical or label-smoothed distribution (row (h-empirical))? The latter is buildable today with no external model and is the lowest-friction route to a non-vacuous outer-loop data term. Fifth, is it acceptable to land the one-line gated-stub comment now, or is outright removal preferred despite the CLAUDE.md pure-path mandate? The recommended action is the former, but it is a one-comment change with no behavioral effect, so the choice is the user's to make.
